# SafeClaw — Risk Assessment

> **Audience:** Security reviewers, operators, contributors.
>
> SafeClaw routes AI tool execution through Docker containers. This document identifies what can go wrong.

---

**Related documents:** [architecture.md](architecture.md) · [requirements.md](requirements.md) · [validance-integration.md](validance-integration.md)

## 1. Why This Matters

An AI agent that can execute arbitrary shell commands, write files, automate browsers, and send messages is inherently dangerous. SafeClaw exists to make this controllable — but "controllable" is not "safe." The safety layer itself has attack surface, failure modes, and limitations that operators must understand.

This document is honest about what SafeClaw protects against and what it does not. If you are running a personal AI assistant that executes code on your machine, you deserve a clear accounting of the risks you are accepting.

| SA-001 | Exec runs as root in container                                | Command runs with uid 0 inside the sandbox container. A container escape exploit has root privileges.                                                                                       | Container isolation (namespaces, cgroups). | SA-004 | In-memory state lost on gateway restart                       | Pending proposals (promises, approval links), session hash cache, and `globalThis` singleton are all in-process memory. Active proposals in the execution engine continue running — only the plugin-side link is lost.                                                       | Users who approved via webhook but haven't retrieved results lose the link. Long-running proposals (5-min exec) in flight at restart time produce orphaned engine-side state.                                                      | Low      | Open   |

No external network exposure in default config.                                                                          | If an attacker has code execution on the host (ironic, given SafeClaw's purpose), they can submit arbitrary proposals to the engine. In a remote deployment, the engine endpoint is fully open. Fix: shared secret / API key header.                              | Medium   | Open   |


## 3. Security Guarantees

What SafeClaw **does** protect against, with the mechanism that provides the protection:


| Guarantee                                            | Mechanism                                                                                          | Evidence                                                             |
| ---------------------------------------------------- | -------------------------------------------------------------------------------------------------- | -------------------------------------------------------------------- |
| LLM cannot execute on the host                       | All dangerous tools are denied in OpenClaw config; SafeClaw routes them through Docker containers  | `DENIED_TOOLS` in `trust-profiles.ts`; user configures `tools.deny`  |
| LLM cannot approve its own actions                   | `/sc-approve` is a plugin command, not a tool. The LLM cannot invoke commands.                     | `api.registerCommand()` in `index.ts`; tool registration is separate |
| Human-confirm actions require explicit user decision | Approval gate blocks the proposal until `/sc-approve` or gateway resolution                        | `approval-handler.ts` resolver; `index.ts` command handler           |
| Actions are rate-limited per session                 | Engine enforces catalog-defined rate limits per `session_hash`                                     | Catalog `rate_limit` field; engine pipeline stage 2                  |
| Secrets never pass through the plugin or LLM         | Engine injects secrets at worker execution time from its secret store                              | `secret_refs` in catalog; engine pipeline stage 5                    |
| Containers have scoped filesystem access             | Volume mounts defined per-template in catalog                                                      | `volumes` field: `/workspace` (rw), `/home` (ro for exec)            |
| Denied actions cannot be called                      | `always-deny` tier actions are excluded from the tool's `action` enum                              | `catalog.ts` `actionNames()` filters out always-deny                 |
| Learned policies are user-created                    | Only `/sc-approve allow-always` and `deny-always` create rules — the LLM cannot trigger `remember` | Decision parsing in `index.ts` command handler                       |


## 4. What SafeClaw Does NOT Protect Against

- **Container escape** — if a vulnerability in Docker or the Linux kernel allows breaking out of the container, the attacker has host access (as root, per SA-001).
- **Engine compromise** — if the execution engine is compromised, all proposals are under attacker control. SafeClaw trusts the engine's responses.
- **Host-level attacker** — anyone with code execution on the host can call the engine API directly (SA-012), read the plugin config (SA-008), or intercept webhooks (SA-007).
- **Supply chain attacks on Docker images** — SafeClaw trusts the Docker images referenced in the catalog. A compromised image can do anything within its container.
- **OpenClaw plugin loader vulnerabilities** — SafeClaw runs inside OpenClaw's process. If the plugin loader has vulnerabilities, SafeClaw inherits them.
- **Multi-user abuse** — SafeClaw is designed for single-user personal assistants. Rate limits and session isolation assume a trusted operator (SA-009).
- **Persistent data exfiltration** — containers with network access (web, comms, browser) can exfiltrate data from mounted volumes. Network policies in the catalog mitigate this partially (`web_search` restricts egress to `*.googleapis.com:443`), but `web_fetch` and `browser` have unrestricted egress.
- **Chrome profile automation** — the `chrome` browser profile (distinct from the `browser` container action) controls the user's existing Chrome browser via extension relay. This cannot be containerized and is explicitly excluded from SafeClaw's scope. Users who need this should NOT deny `browser` in their config.
